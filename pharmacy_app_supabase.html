🕓 Time Dependency

Level 2 temporal tracking — who changed what and when

This database implements Level 2 Time Dependency. Tables track created_at, updated_at, and updated_by. When a prescription is dispensed, updated_at = NOW() and updated_by = 'Pharmacist' are recorded automatically.

∑ Relational Algebra

Assignment queries — live results from Supabase (PostgreSQL)

Query 1 — Dispensed Prescriptions
πDoctorName, CustomerName, PharmacistName, PrescribedDate
(σstatus='Dispensed'(PRESCRIPTION) ⋈ DOCTOR ⋈ PERSON ⋈ CUSTOMER ⋈ PHARMACIST)
Query 2 — Revenue by Payment Method
πpayment_method, COUNT(*), SUM(amount_paid)
(σstatus='Completed'(PAYMENT ⋈ INVOICE)) GROUP BY payment_method
Query 3 — Low Stock Medicines
πMedicineName, CurrentStock, ReorderLevel, Supplier
(σquantity < monitor_level(INVENTORY ⋈ MEDICINE ⋈ PURCHASE_ORDER ⋈ SUPPLIER))
Query 4 — Set Operation: INTERSECT (Customers who Dispensed AND Paid)
Relational Algebra — Set Operation (Intersection concept)
Finds customers who appear in both the set of dispensed prescriptions AND the set of completed payments — demonstrating the INTERSECT operation taught in class.
πCustomerName(σstatus='Dispensed'(PRESCRIPTION ⋈ CUSTOMER ⋈ PERSON))
∩
πCustomerName(σpayment_status='Completed'(PAYMENT ⋈ INVOICE ⋈ PRESCRIPTION ⋈ CUSTOMER ⋈ PERSON))
Normalization Reference — All Tables Satisfy 3NF
Normal FormRuleHow This DB Satisfies ItStatus
1NF Every column must be atomic (no multivalued attributes) No column stores multiple values. Skills, medicines etc. are stored in separate rows in bridge tables like PRESC_MED_DETAIL ✓ Satisfied
2NF No partial dependency — non-key attributes must depend on the whole primary key All tables use surrogate PKs (SERIAL). PRESC_MED_DETAIL has composite PK (prescription_id, medicine_id) and all its attributes (quantity, dosage_instructions) depend on both parts together ✓ Satisfied
3NF No transitive dependency — non-key attributes must not depend on other non-key attributes Example: PERSON stores name/phone/email. DOCTOR stores only doctor-specific data referencing PERSON. No column determines another non-key column transitively ✓ Satisfied
